KNOWN FALSE POSITIVES OBSERVED IN THE RUN
================================================================

FP1  Partial-class collision in top-level Program.cs
     [::Program]  Catalog.API/Program.Testing.cs <-> Ordering.API/Program.Testing.cs
     Both files declare `public partial class Program { }` with NO
     namespace (top-level statements). The grouping key becomes "::Program"
     and unrelated partials from different projects collide.
     Fix (added to spec): include project root (nearest *.csproj dir)
     in the partial-class grouping key.

FP2  EF-nav ambiguity from duplicate short names
     `Order.OrderItems: OrderItem` resolves to TWO files:
       - src/ClientApp/Models/Orders/OrderItem.cs
       - src/Ordering.Domain/AggregatesModel/OrderAggregate/OrderItem.cs
     Both exist (one a DTO mirror of the other). Both edges emitted.
     This is acceptable — neighbor signals are robust to multi-target,
     and the spec's "prefer same-namespace-prefix" tiebreaker (not
     implemented in the POC) would pick the correct one.

FP3  Nav-edge `CustomerBasket.Items: BasketItem` resolves to FIVE files.
     Same root cause as FP2 — `BasketItem` is a common DTO name. Five
     copies (Basket.API model, ClientApp model, gRPC proto generated,
     Ordering.API model, WebApp model). Spec calls for emitting all
     candidates by default; production rollout should add the
     same-namespace-prefix tiebreaker.

================================================================
TAKEAWAYS
================================================================

- All four convention types parse cleanly with tree-sitter-c-sharp 0.21.
  Zero parse errors across 637 files.
- Partial-class detection is the cheapest win and has near-zero FPs
  with the namespace-key fix (FP1).
- DI registrations resolve ~46% on raw short-name lookup; remaining
  unresolved cases are mostly cross-package types (framework interfaces
  like ICacheClient, IConnectionMultiplexer that live in NuGet packages)
  — correctly left unresolved and would be stoplisted in production.
- EF nav coverage is 68%; unresolved are again framework types or
  primitives the gate doesn't catch.
- Attribute routing extracts cleanly; metadata-only as per spec
  (no edges generated, recorded on the controller symbol for surfacing
  through get-structure).

The POC is intentionally minimal — no project-root awareness, no
stoplist, no same-namespace tiebreaker. All of those are described in
spec.md and would land alongside production integration.
