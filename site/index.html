Codebase memory & navigation for AI agents

Your agent lands cold in every repo.

Two layers, one tool

Exact structure, plus honest memory.

The cold layer keeps facts exact. The warm layer keeps judgment honest. The agent supplies both — you maintain neither.

❄ Navigation · the index

Find the file. Fast.

A static index over paths, symbols, exports, and the import/call graph. Deterministic, offline, free — no API key, no GPU, no inference.

find <terms>

Ranks files by how many of your query terms each covers — and shows why it ranked: which terms it defines vs. imports.

gs <file>

One file's shape: symbols with line ranges, who imports it, who calls each symbol. Not a grep — the answer to "who uses this?"

The notebook · written and maintained by the agent

A memory your agent keeps — so you don't have to.

Most "knowledge bases" rot because a human has to feed them. coldstart's notebook is written by the agent that just did the work, kept honest by the index automatically, and recalled the moment it's relevant. There is no human in the maintenance loop — and that's the point.

1The agent writes it

Authored after a real task.

The reasoner that just solved something writes down what it learned — a file's purpose, a cross-file flow, a trap — with concrete citations, while the full context is still in hand. A write gate dedupes by concept, and concurrent agents never silently merge or lose a note.

You write the docs.  The agent does.

Shared, it compounds faster

Commit the notebook to your repo and the whole team feeds one corpus. The append-only logs union-merge across branches and machines — no conflicts — so every teammate's agent both reads from and adds to the same notebook. Private by default; opt in with coldstart init --commit-notebook. More sessions in, bigger notebook out.

A knowledge base that compounds as the repo gets worked — more context for the next agent, less effort spent re-deriving what's already known.

The intended flow

find → gs → Read.

Orient with two cheap calls before you spend a single token reading. Notebook summaries ride along on find, so the orientation step often answers itself.

Bring your own semantics

No embeddings. On purpose.

Every consumer is already a frontier model — the best semantic reasoner in the loop. Pre-computing meaning at index time only duplicates that, worse and stale. So coldstart keeps what's cheap to keep exact, and leaves meaning to the reasoner holding the task.

Embedding-based code search

  • ✕Freezes one interpretation of a file at index time, blind to the task.
  • ✕A stale vector fails silently — a confident wrong answer you can't spot.
  • ✕Returns a cosine score the agent has to trust blindly or re-verify.
  • ✕Needs an API key, a GPU, inference on every query.

coldstart

  • ✓The agent judges relevance in context, against the actual goal.
  • ✓A background keeper patches facts in milliseconds — the index never lies about the tree.
  • ✓Returns checkable evidence — this term is defined here, imported there.
  • ✓Runs on the agent you already pay for — deterministic, offline, no key, no GPU.
No second bill

coldstart runs on the agent you already pay for. The semantic layer is your Claude Code, Codex, or Cursor subscription — coldstart adds no inference of its own, so there's no separate API key, no metered embeddings, nothing new to provision. Most code-search tools bill you twice: once for the agent, again for their own model. This one doesn't.

Get started

Two commands to warm the repo.

Works best with Claude Code, Codex, and Cursor — all get the fast CLI path and the notebook capture/recall hooks auto-wired by init.

your-project
$ npm install -g @cstart/coldstart --legacy-peer-deps $ cd your-project $ coldstart init # coldstart.md + client wiring + notebook + capture/recall hooks ✓ index warming in the background — your first lookup is instant.
